## Design Context

### Users
Mixed internal teams — cross-functional audiences consuming dashboards built with this library. Includes technical authors (analysts / engineers wiring up Rust + Polars data pipelines) and non-technical consumers (business stakeholders scanning the generated HTML reports). Viewing contexts span desk monitors, shared conference-room screens, and occasional laptop scroll-throughs. Users need both fast scannability ("what's the top-line number?") and drill-down interactivity (range sliders, linked selection, date filters, RangeTool navigators). The `example-dashboard` binary doubles as library marketing — developers evaluating the project judge it by how the output looks.

### Brand Personality
**Technical · Industrial · Utilitarian.** Think lab instrument, control panel, scientific workbench — not consumer app, not SaaS marketing page. Voice: precise, unadorned, trustworthy. The data is the hero; the UI is scaffolding around it. Confident without being flashy; dense without being cramped. Functional beauty — the aesthetic of a well-machined tool.

### Aesthetic Direction
- **Reference points:** observability dashboards with less chrome (Grafana stripped back), engineering CAD tooling, scientific instrument UIs, IBM Carbon with a sharper personality, Bloomberg-terminal information density paired with modern typography.
- **Theme:** Both light and dark, user-switchable. Respect `prefers-color-scheme` on first load; remember user toggle after. Every color token must hold up in both themes — no dark-mode afterthought.
- **Color strategy:** OKLCH-based palette, neutrals tinted toward a single brand hue, one sharp functional accent used sparingly (alerts, active states, selected filters). No gradient text. No glow effects. No glassmorphism.
- **Typography:** Humanist body font for prose (paragraphs, labels), tabular / monospace-leaning face for numeric data so columns align naturally. Avoid the reflex AI picks (Inter, IBM Plex, Space Grotesk, DM Sans, Plus Jakarta, Instrument Serif). Look at foundries like Klim, ABC Dinamo, Pangram Pangram, Velvetyne for the display face.
- **Layout:** Dense but rhythmic. Varied spacing for hierarchy. Flat cards — no nested containers, no shadowy floating panels. Grid-first; break the grid only when a chart earns it.
- **Anti-patterns to avoid:** generic blue-sidebar-plus-white-cards SaaS template (roughly the current state of `templates/chart.html`), border-left accent stripes on cards, uniform card grids with matching icons, rounded-rectangle-plus-drop-shadow safe styling, cyan-on-black neon, purple-to-blue gradients.

### Design Principles
1. **Data over chrome.** Charts, tables, and numbers are the focal elements. Nav, filters, and cards are scaffolding — they should recede visually and never compete with content.
2. **Honest hierarchy through spacing, not decoration.** Variable gap, type scale, and weight communicate importance. Avoid the uniform-card-grid trap that flattens every element to the same visual weight.
3. **Dual-theme discipline.** Every decision — color, shadow, border, tinted neutral — holds up in both light and dark. Use OKLCH + `light-dark()` or explicit theme tokens. Never hardcode hex values in the final output.
4. **Tabular numerals for comparison, humanist type for prose.** Pair a numerically-disciplined face (tabular figures, monospaced digits) with a readable body font. Pick a display face that reinforces the industrial character — not the standard AI-default list.
5. **Functional motion only.** Transitions serve state changes: filter applied, page switched, row selected. No decorative bounce, no scroll-reveals, no elastic easing. A reporting tool should feel responsive, not performative.
